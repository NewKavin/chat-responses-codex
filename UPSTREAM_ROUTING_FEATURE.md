# 上游账号限制显示与智能路由功能

## 功能概述

本次更新实现了上游账号的限制信息显示和智能路由算法,用于保护高端模型账号的额度,避免被低权重模型占用。

## 主要功能

### 1. 上游账号限制显示 (仅显示,不做实际校验)

前端界面显示以下上游账号信息:
- **每分钟请求数**: 显示当前使用量和限制,带进度条
- **5小时请求配额**: 显示当前使用量和限制,带进度条
- **最大并发数**: 显示当前并发数和最大并发限制
- **高端模型保护状态**: 显示是否启用保护及保护的模型数量

这些限制信息仅用于监控和显示,后端不做强制校验。

### 2. 智能路由算法

#### 核心逻辑

路由算法实现了以下智能调度策略:

1. **优先级分组**: 将上游账号分为"首选组"和"回退组"
   - 首选组: 未启用高端模型保护的账号,或请求的模型在其高端模型列表中
   - 回退组: 启用了高端模型保护且请求的模型不在其高端模型列表中

2. **首选组优先**: 优先从首选组中选择账号,避免占用高端账号的额度

3. **回退机制**: 仅当首选组无可用账号时,才使用回退组的账号

4. **排序规则**: 在每个组内,按以下顺序排序:
   - 优先级权重 (高到低)
   - 失败次数 (低到高)

#### 使用场景示例

**场景 1: 保护高端模型账号**
```
配置:
- 账号 A: 支持 [gpt-4, gpt-3.5-turbo, glm-5.1]
  - 高端模型: [glm-5.1]
  - 保护高端额度: 开启
  - 优先级: 100

- 账号 B: 支持 [gpt-4, gpt-3.5-turbo]
  - 优先级: 50

请求: gpt-4 (非高端模型)
结果: 选择账号 B,即使账号 A 优先级更高
原因: 避免占用账号 A 的高端模型额度
```

**场景 2: 高端模型路由**
```
请求: glm-5.1 (高端模型)
结果: 选择账号 A
原因: glm-5.1 在账号 A 的高端模型列表中,应该使用该账号
```

**场景 3: 回退机制**
```
配置: 只有账号 A 可用
请求: gpt-4 (非高端模型)
结果: 使用账号 A 作为回退
原因: 没有其他可用账号时,仍然可以使用保护账号
```

### 3. 前端配置界面

#### 限额配置区域
- 每分钟请求数
- 5小时请求配额
- 最大并发数
- 说明: "用于显示和监控,不做强制限制"

#### 智能路由配置区域
- **优先级权重**: 数字越大优先级越高
- **高端模型列表**: 选择此账号独有的高端模型
- **保护高端额度**: 开关,开启后非高端模型会优先避开此账号

#### 实时状态显示
- 表格中显示每个上游的实时运行状态
- 进度条颜色:
  - 绿色 (< 60%): 正常
  - 橙色 (60-80%): 警告
  - 红色 (> 80%): 高负载
- 自动刷新: 每5秒更新一次

## 配置示例

### 典型配置: 保护 GLM-5.1 高端账号

```json
{
  "id": "premium-glm",
  "name": "GLM 高端账号",
  "base_url": "https://api.example.com",
  "api_key": "sk-xxx",
  "protocol": "ChatCompletions",
  "supported_models": ["gpt-4", "gpt-3.5-turbo", "glm-5.1", "deepseek-chat"],
  "requests_per_minute": 100,
  "request_quota_5h": 1000,
  "max_concurrency": 10,
  "priority": 100,
  "premium_models": ["glm-5.1"],
  "protect_premium_quota": true,
  "active": true
}
```

### 配置说明

- `premium_models`: 列出此账号独有的高端模型
- `protect_premium_quota`: 设置为 `true` 启用保护
- `priority`: 高优先级确保高端模型优先使用此账号
- 当请求非高端模型(如 gpt-4)时,系统会优先选择其他账号

## 使用建议

1. **识别高端账号**: 找出支持独特高端模型的账号(如 glm-5.1)
2. **配置高端模型列表**: 在该账号的 `premium_models` 中列出高端模型
3. **启用保护**: 开启 `protect_premium_quota` 开关
4. **设置高优先级**: 确保高端账号有较高的 `priority` 值
5. **监控使用情况**: 通过前端界面实时查看各账号的负载情况

## 注意事项

- 限额配置仅用于显示,不做强制限制
- 保护机制是软性的,当无其他选项时仍会使用保护账号
- 建议定期查看运行时状态,了解各账号的实际使用情况
- 自动刷新每5秒更新一次,可能会增加服务器负载
