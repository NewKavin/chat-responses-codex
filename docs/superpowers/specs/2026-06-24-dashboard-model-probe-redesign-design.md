# 控制台总览与模型探测页重设计方案

**目标：** 将管理端控制台总览重做为更高密度、更少文字、更强统计感的企业后台页面，并在管理员与门户各新增一个自动刷新模型探测页，展示渠道健康、模型覆盖和最近探测结果。

## 范围

- 重做 `/admin/dashboard` 的视觉层级与图表组合。
- 为管理员新增一个“模型探测”页面。
- 为门户新增一个同主题的“模型探测”页面，内容按当前门户可见范围过滤。
- 前端只使用本地打包依赖，不引入任何外网脚本或 CDN。

## 设计方向

页面气质采用方案 A 的“中台总览”风格：

- 顶部用少量 KPI 卡片承接核心指标。
- 中部用 4 到 6 个图表卡片构成主视觉。
- 底部只保留少量排名、状态和说明，不再堆长段文字。
- 图表以 ECharts 为主，优先使用饼图、环形图、柱状图、折线图和轻量状态卡片。

## 管理端总览

管理端总览的关键信息分成四层：

1. 业务总量：上游数、下游数、日志量、可见模型数。
2. 结构分布：模型使用饼图、下游客户端柱状图或饼图、失败分类饼图。
3. 趋势走势：请求量、成功率、平均延迟、Token 总量的时间序列。
4. 运维信号：User-Agent 聚类、Responses 上游占比、活跃通道概览。

页面会减少解释性段落，把原本占位的说明文本换成图表和紧凑状态卡。

## 模型探测页

模型探测页参考 sub2api 的渠道状态视图，但按本项目的数据结构做两层展示：

- 渠道层：每个上游或 API key 的健康状态、探测耗时、模型数、最近探测时间、失败原因。
- 模型层：每个模型的可用渠道数、可见状态、是否被当前范围覆盖。

页面只做自动刷新，不提供手动探测按钮。自动刷新由前端轮询控制，后端返回当前最新快照并使用短时缓存避免过度探测。

门户版本只展示当前门户可见模型与相关渠道汇总，不暴露不必要的管理信息。

## 数据与接口

- 扩展 `/api/admin/dashboard`，补充模型使用、客户端分布等聚合统计。
- 新增 `/api/admin/model-probe`，返回管理员范围内的渠道与模型探测快照。
- 新增 `/api/portal/model-probe`，返回当前门户范围内的探测快照。

所有数据均来自本地服务端计算与本地打包前端，不依赖公网。

## 非目标

- 不做手动探测按钮。
- 不做第三方分析平台接入。
- 不改变现有认证模型。
- 不引入新的外部前端 CDN。

## 成功标准

- 管理端总览页面在视觉上以图表为主，文字说明明显减少。
- 模型探测页可自动刷新并显示渠道状态与模型覆盖。
- 管理端与门户都能访问各自的模型探测页。
- 前端构建和测试在离线环境下通过。
