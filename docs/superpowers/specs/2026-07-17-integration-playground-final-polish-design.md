# 集成示例与模型操练场最终整理设计

## 背景

门户集成示例已经能从当前下游密钥、网关地址和实时模型目录生成 Codex、OpenCode、
Claude Code、OpenAI、Anthropic 与 Hermes Agent 配置，但页面顶部状态、协议说明、模型推荐和
代码示例连续展开，扫描路径偏长。模型操练场的参数开关使用 Element Plus inline prompt，浅色
主题下未选中的白色“自动”文字叠在浅灰轨道上，对比度不足。

本轮作为最终部署前的窄范围整理，不改变协议转换、模型目录、配置生成或上游调度行为。

## 目标

- 集成示例保持工作台式信息密度，同时让连接状态、推荐模型和客户端配置的层级更清楚。
- 客户端示例按常用度排列，Codex 与 OpenCode 保持最优先。
- 保留现有实时目录校验、空状态、告警、复制和 doctor 指引。
- 提高模型操练场未选中“自动”文字在浅色主题下的可读性。
- 最终部署后用真实 Codex 和 OpenCode 客户端验证指定的八类常用模型。

## 非目标

- 不修改门户 API、后端模型排序、下游密钥或配置生成器的数据契约。
- 不增加新的客户端、协议说明或营销内容。
- 不改变密钥展示与复制行为。
- 不改变 Element Plus 全局主题或其他页面的开关样式。
- 不对模型输出质量做主观排名，也不对全部上游模型做穷举测试。

## 集成示例信息结构

页面继续使用现有扁平分区和设计 token，不增加嵌套卡片。

1. 顶部连接摘要保留网关地址、当前 key 状态、模型数和默认模型，但收紧标题、间距和字段布局，
   使其成为可快速核对的状态条。现有 warning/error 仍独立占行，不能被折叠隐藏。
2. 协议兼容矩阵保留 Responses、Chat Completions 和 Anthropic Messages 三类入口，保持 endpoint、
   鉴权和适用客户端可直接扫描。
3. 推荐模型从松散 chip 网格整理为紧凑、可换行的排名列表，继续显示月调用、今日调用和默认模型
   标识；长 slug 必须完整可读且不能撑宽移动端页面。
4. 示例标签按 Codex、OpenCode、Claude Code、Cline / OpenAI、Anthropic、Hermes Agent 排列。
   每个标签内部沿用现有步骤、代码块、复制按钮和实时模型配置，不改变生成内容。

桌面端保持紧凑多列摘要，窄屏降为单列。固定格式的代码区继续使用有界高度和自身滚动，不扩大
页面文档宽度。

## 模型操练场对比度

根因是 Element Plus inline prompt 固定使用白色文字，而浅色主题的未选中轨道使用浅边框色。
修复只作用于 `PlaygroundSettings` 内未选中的 inline switch：将其内部文字改为
`var(--crc-text-strong)`。选中状态继续使用白字和强调色背景，其他页面开关不受影响。

## 测试与验收

自动化验证覆盖：

- 集成示例的区块顺序、客户端标签顺序和紧凑模型列表结构；
- 原有实时目录、空状态、复制按钮、Codex `client_version=0.144.4` 与配置字段仍存在；
- 操练场三个“自动”提示保留，未选中状态使用局部高对比选择器；
- 前端完整测试和生产构建；
- Rust 回归测试与 release 构建。

浏览器验收覆盖桌面与移动端：无文本重叠、无页面级横向溢出、代码可复制、长模型 slug 可读，
以及“自动”在浅色主题下清楚可见。

部署后通过真实 Codex `0.144.4` 和 OpenCode `1.17.18` 对以下 exposed slug 分别执行有实质内容的
协议分析任务与只读文件工具任务，不发送 `hi`、`hello` 或“你好”探活：

- `glm-5.2`
- `deepseek-v4-flash`
- `deepseek-v4-pro`
- `Qwen/Qwen3.5-122B-A10B`
- `MiniMax-M2.7`
- `kimi-k2.5`
- `grok-4.5`
- `claude-opus-4-7`

每次记录客户端、模型、任务类型、退出状态和耗时；结合网关日志核对 499、502、panic、ERROR、
容器重启、首个有意义事件耗时和最终上游。单模型失败单独重试一次以区分稳定兼容问题与上游偶发
晚流错误，不在已经输出有效内容后切换上游并混合响应。
