# 控制台全宽布局修正设计

## 背景

当前所有管理端和门户页面都以 `.crc-page` 作为根容器。共享样式为该容器设置了
`max-width: 1600px` 和 `margin: 0 auto`，因此在超宽屏上只显示为居中的内容条带，
没有利用侧栏之外的可用空间。

日志管理受到同一全局限制。排障中心还把 Capability 宽表放进约三分之一宽度的左栏，
同时兼容矩阵使用较大的表格最小宽度，造成多层横向滚动和混乱的信息层级。

## 目标

- 所有业务页面铺满 AppShell 内容区，保留统一的页面内边距。
- 日志管理的筛选、统计、表格和分页使用完整内容区宽度。
- 排障中心按照日常工作流重新组织配置、结果、Capability 和兼容矩阵。
- 宽表只在自身表格容器内横向滚动，不触发页面级横向滚动。
- 桌面、中等宽度和移动端均保持清楚、稳定的布局。

## 非目标

- 不修改 API、状态管理、诊断流程或日志字段。
- 不改变登录卡片、长文本、聊天输入等局部可读宽度约束。
- 不删除兼容矩阵或日志表格字段。
- 不重新设计配色、导航或组件视觉语言。

## 共享页面宽度

`.crc-page` 保留 `width: 100%`，移除 `max-width: 1600px`。AppShell 继续负责桌面和
移动端的内容区 padding，因此页面会铺满可用空间，但不会贴紧视口边缘。

局部 `max-width` 继续保留，包括：

- 登录面板；
- 页面说明文字；
- 公告编辑表面；
- 密钥安全设置；
- Playground 输入区。

这些约束服务于表单或文本可读性，不属于全局页面限宽。

## 日志管理

日志页继续沿用现有信息顺序：

1. 页面标题和刷新操作；
2. 筛选条件与快捷分类；
3. 提示和统计摘要；
4. 日志表格；
5. 分页。

页面根容器铺满 AppShell 内容区。日志表格外层保持 `width: 100%` 和
`overflow-x: auto`，表格的宽字段仍可横向滚动，但滚动范围限制在表格表面内。
筛选工具栏在桌面充分利用横向空间，在移动端保持单列输入和可横向滚动的快捷筛选。

## 排障中心

采用用户确认的“工作流优先”布局。

### 诊断工作区

诊断工作区采用自适应 CSS Grid，不写死页面宽度或固定百分比。空间充足时使用类似以下
轨道策略：

```css
grid-template-columns:
  minmax(320px, 0.75fr)
  minmax(560px, 1.25fr);
```

两栏根据内容区实际宽度伸缩：

- 左栏只放诊断配置、下游、客户端、模型和诊断项目；
- 右栏放诊断结果和活跃任务；
- 结果标题和操作栏允许换行，操作按钮不被压缩。

Capability 策略不再放进左侧窄栏。

### Capability 策略

Capability 操作、profiles 表、resolved capability 表和 conflicts 表移动到诊断工作区
下方的全宽 section。所有表格共享一层横向滚动容器，避免窄栏内嵌宽表。

### 兼容矩阵

兼容矩阵继续位于页面底部，使用完整内容区宽度。矩阵和展开详情分别在自身表格容器中
滚动，不扩大页面文档宽度。工具栏允许换行，并在中等宽度提前切换为纵向排列。

## 响应式规则

- 宽屏桌面：诊断配置和结果使用 `minmax()` 与弹性比例自动分配空间。
- 当诊断容器无法同时满足两栏最小可用宽度时，通过容器查询切换为单列；Capability 与
  兼容矩阵始终保持全宽。
- 移动端：所有 section 单列；输入控件和主操作占满可用宽度；快捷筛选、分页和宽表仅在
  自身容器内横向滚动。
- 所有断点下，`document.documentElement.scrollWidth` 不应超过 viewport 宽度。

## 测试与验收

自动化契约覆盖：

- `.crc-page` 不再声明全局 `max-width`；
- Capability 区域位于诊断两栏之外；
- 诊断工作区使用自适应 Grid，并在容器空间不足时切换为单列；
- 排障工具栏具备换行与中等宽度响应式规则；
- 日志页继续使用共享全宽表格容器。

浏览器验证覆盖：

- `2560x1440` 浅色桌面；
- `1440x900` 深色桌面；
- `390x844` 深色移动端；
- 日志管理、排障中心，以及至少一个管理端和一个门户普通页面；
- 页面级横向溢出、控件重叠、标题挤压和内部表格滚动。
