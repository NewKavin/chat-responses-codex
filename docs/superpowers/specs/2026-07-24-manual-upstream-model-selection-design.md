# 上游模型手动选择设计

## 状态

本设计已确认采用全局开关和手动候选选择方案。它收紧
`2026-07-18-key-model-route-resilience-design.md` 中后台模型发现的启用条件，但不改变
精确路由健康、429 fallback、cooldown、half-open 或终态错误语义。

## 问题

管理端当前存在两条会把上游 `/v1/models` 结果直接写入路由目录的路径：

- 创建多 Key 上游时，后端无条件发现模型并保存全部结果；
- 编辑上游时，“获取模型”会用发现并集直接覆盖 `supported_models`。

后台模型-Key 同步还会按 `UPSTREAM_MODEL_KEY_SYNC_INTERVAL_SECONDS` 周期更新映射。这样
上游公布但管理员不想使用、不能稳定调用或不适合 Codex 的模型会自动进入 `/v1/models`
和生产路由。它们发生 429 后会按现有设计切换候选并进入精确路由 cooldown；所有候选都
冷却时，Codex 收到 `503 upstream_routes_exhausted` 和
`all eligible upstream routes are temporarily unavailable`。

## 目标

- “获取模型”只加载候选项，不自动选中或持久化模型。
- 管理员明确选中并保存的模型才进入 `supported_models`、`api_key_models` 和生产路由。
- 增加默认关闭的全局开关，统一控制会写回配置的自动模型发现。
- 保留多 Key 的精确模型映射，不因账号级手动选择退化为盲试所有 Key。
- 保持已有配置不变；升级本身不自动删除现存模型。
- 保持现有 429 fallback 和 route-health 行为。

## 非目标

- 不增加逐上游自动发现开关或数据库字段。
- 不让 route cooldown 修改持久模型目录。
- 不在网关请求内等待上游 `Retry-After` 后重放 Codex 请求。
- 不改变模型健康页的只读探测和浏览器刷新行为。
- 不保证管理员选中的模型永不被上游限流；生产可用性仍需要健康的备用路由。

## 方案选择

### 采用：全局硬开关加手动候选选择

新增 `UPSTREAM_MODEL_AUTO_DISCOVERY_ENABLED`。默认值为 `false`。关闭时，创建上游、
周期同步和模型不匹配触发的定向发现都不能自动发现并写回模型；显式管理端发现接口始终
可用。管理员点击“获取模型”后从候选项中选择需要的模型，再通过保存动作确认目录。

该方案同时覆盖创建和后台同步，且无需数据库迁移。它保留显式发现的便利性，也让管理员
选择成为持久目录的唯一默认写入来源。

### 未选：只使用同步间隔 `0`

`UPSTREAM_MODEL_KEY_SYNC_INTERVAL_SECONDS=0` 只能关闭后台服务，不能表达创建上游时是否
自动发现，也不能提供清晰的布尔开关语义。

### 未选：逐上游开关

逐上游开关更灵活，但需要配置模型、PostgreSQL、文件存储、API 和 UI 全链路迁移。当前
需求是默认采用统一的手动管理策略，额外复杂度没有必要。

## 配置语义

新增环境变量：

```text
UPSTREAM_MODEL_AUTO_DISCOVERY_ENABLED=false
```

配置真值表：

| 自动发现开关 | 同步间隔 | 创建时自动发现 | 周期/定向发现 | 手工“获取模型” |
|---|---:|---|---|---|
| `false` | 任意值 | 关闭 | 关闭 | 可用 |
| `true` | `0` | 开启 | 关闭 | 可用 |
| `true` | 正整数 | 开启 | 开启 | 可用 |

自动发现开关只控制会写回 `api_key_models`、`supported_models` 或 `last_synced_at` 的入口。
管理员/门户模型健康页对 `/v1/models` 的只读探测继续由现有刷新和缓存配置控制。

`.env.example`、Docker Compose、README 和部署文档必须把新开关记录为默认 `false`。现有
`UPSTREAM_MODEL_KEY_SYNC_INTERVAL_SECONDS` 继续只表示启用后的周期频率。

## 管理端交互

### 获取候选

上游编辑器维护两类独立状态：

- `supported_models`：管理员当前选中的权威模型集合；
- discovery candidates：最近一次显式发现得到的临时候选和逐 Key 结果。

点击“获取模型”后：

1. 按当前 Base URL 和 Key 调用显式发现接口；
2. 将成功结果加入下拉候选项；
3. 保留当前已选模型，不新增选中项，也不移除任何选中项；
4. 展示成功数量和逐 Key 安全错误摘要；
5. 在关闭抽屉或重新载入页面时丢弃未保存的候选状态。

候选下拉继续允许手工输入，以支持 `/v1/models` 未列出但管理员确认可调用的模型。

### 保存选择

保存前根据当前 Key、管理员选择、已有映射和本次发现结果构造权威映射：

- 取消选择的模型从所有 Key 映射中移除；
- 本次发现到的已选模型只保留在实际发现它的 Key 上；
- 某个 Key 发现失败时，已选模型可以保留该 Key 的最后成功映射；
- 管理员手工输入且没有任何发现证据的模型视为显式断言，映射到当前所有 Key；
- 每个当前 Key 都保留一条映射，空列表表示该 Key 未获准参与任何模型路由；
- `supported_models` 由最终 Key 映射并集得到，并必须等于管理员最终选择。

创建和编辑使用同一映射构造规则。批量创建 API 需要接受显式
`supported_models`/`api_key_models`；自动发现关闭时不能发起上游模型请求，而应直接保存
管理员提交的规范化映射。

## 自动发现开启时的兼容行为

显式把 `UPSTREAM_MODEL_AUTO_DISCOVERY_ENABLED=true` 视为管理员选择恢复旧的自动管理
模式：创建上游可以自动发现并保存结果；同步间隔为正数时，周期和定向发现可以按既有
安全写回规则更新目录。手工按钮仍采用候选模式，避免一个交互动作绕过用户选择。

自动模式不能改变以下既有不变量：失败发现保留旧映射、空发现不建立权威映射、删除 Key
不能被旧任务恢复、运行时 cooldown 不能改写模型目录。

## 已有数据和迁移

本次不迁移或重写已有上游记录。升级后：

- 现存 `supported_models` 和 `api_key_models` 原样加载；
- 默认关闭自动发现后，后台不再新增或删除模型；
- 管理员打开编辑器时看到已有选择；只有取消选择并保存才删除模型；
- 配置为自动模式的部署必须显式设置新开关为 `true`。

## 429 和路由耗尽边界

本设计不修改现有尝试顺序：

1. 上游 429 不在原 route 内等待或重试；
2. 网关按上游 `Retry-After` 冷却 exact route，并立即尝试下一个 Key/上游/协议；
3. 全部候选临时失败或冷却时返回 `503 upstream_routes_exhausted`，响应使用最早
   `Retry-After`；
4. cooldown 到期后只允许一个 half-open 请求探测，成功清除状态，失败重新冷却；
5. 普通 5xx/transport 仍按现有规则在同 route 短退避重试一次后 fallback。

手动模型选择的作用是阻止未授权或不稳定模型自动进入候选集合，不是绕过供应商限流。

## 错误处理

- 手工发现全部失败时不改变当前选择和已有映射。
- 部分 Key 失败时显示失败索引；成功 Key 的候选仍可选择。
- 保存结果为空是合法的权威配置；该上游不会广告或路由任何模型。
- 自动开关关闭时，后台服务不得悄悄启动或写回配置。
- 新配置解析失败沿用现有环境变量布尔解析和启动错误策略。

## 测试设计

### 配置和后台服务

- 新开关缺失时默认为 `false`，显式 `true` 能启用；
- `false` 加任意同步间隔都不启动周期或定向发现服务；
- `true` 加间隔 `0` 只允许创建时自动发现；
- `true` 加正间隔保持现有后台同步行为。

### 后端管理 API

- 自动关闭的批量创建不会请求上游 `/v1/models`；
- 自动关闭时精确保留提交的模型和逐 Key 映射；
- 自动开启时保留既有安全发现和写回语义；
- 显式 discover endpoint 不受全局开关影响；
- 规范化拒绝旧 Key 映射并保持 authoritative 空项。

### 前端

- 手工获取成功只增加候选，不改变已选模型；
- 重复获取不会重复候选；
- 取消选择后，提交映射不再包含该模型；
- 分 Key 发现结果只映射到支持该模型的 Key；
- 手工输入的未知模型映射到所有当前 Key；
- 部分和全部发现失败都保留当前选择。

### Gateway 回归

- 未选择的发现模型不进入 `/v1/models`，也不生成生产候选路由；
- 已选择模型继续使用精确 Key 映射和 fallback；
- 429 仍切换其他健康候选，全部 cooldown 时仍返回 503 和最短 `Retry-After`；
- cooldown/half-open 不修改持久目录。

## 验收标准

1. 默认部署不会自动发现并加入上游模型。
2. 点击“获取模型”后，模型只作为候选出现，选中集合保持不变。
3. 保存后只有管理员选中的模型出现在上游配置和 `/v1/models`。
4. 多 Key 上游仍只把模型发给已确认支持它的 Key。
5. 自动发现只有显式设置新开关为 `true` 才能运行。
6. 已有配置升级后不发生自动删除或新增。
7. 429 fallback、cooldown、half-open 和 503 `Retry-After` 行为不回归。
8. Rust 测试、前端测试、类型检查和构建全部通过。
