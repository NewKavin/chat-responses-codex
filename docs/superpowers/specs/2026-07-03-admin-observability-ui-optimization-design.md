# 管理台可观测性页面优化设计

## 背景

当前网关已经完成协议兼容、错误可见性和模型探测基础能力。用户主要通过 codex、opencode、claude code、Cline 插件、Hermes agent 等客户端长时间连续调用网关，因此管理台需要更快回答三个问题：

1. 某个模型或通道当前是否健康。
2. 请求失败是配额、限流、上游协议、流式中断还是网关自身拦截。
3. 模型操练场在模型慢、空返回或报错时，是否能明确告诉操作者当前状态。

本次优化聚焦用户确认的 A、B、D 三个方向：

- A：模型操练场体验。
- B：日志定位与错误分类。
- D：概览与模型探测。

## 目标

- 让模型操练场在等待首包、推理中、生成中、空返回、错误返回时都有明确反馈，避免用户误以为页面卡死。
- 让日志页更适合排障，突出错误分类、模型、客户端、上游和状态码之间的关系。
- 让 Dashboard 提供模型健康入口，并与模型探测页形成巡检链路。
- 修正模型探测页空数据和错误状态无法正确区分的问题。
- 保持现有 API 协议不变，不猜测模型厂商私有协议。

## 非目标

- 不修改 Chat Completions、Responses、Claude、工具调用等网关转换协议。
- 不修改上游模型适配规则、参数清洗策略或路由策略。
- 不重做全局导航、登录页或整体主题。
- 不引入新的后端接口，除非实现阶段发现现有接口无法完成最低可用体验；这种情况需要单独确认。

## 方案选择

推荐方案是“小步联动优化”：同时优化操练场、日志页、Dashboard 和模型探测页，但每个页面只做排障效率相关的低风险改动。

备选方案一是只做模型探测页，风险最低，但无法解决操练场空返回和日志定位慢的问题。

备选方案二是重做 A/B/D 页面结构，视觉变化更明显，但改动面大、回归成本高，容易影响当前管理台使用习惯。

本设计采用推荐方案。

## 页面设计

### 模型操练场

文件范围：

- `frontend/src/views/portal/Playground.vue`
- `frontend/src/utils/playground.ts`

操练场保留现有左右布局和输入区，只增强运行状态和结果解释。

新增或强化的信息：

- 请求阶段：连接中、等待首包、思考中、生成中、已完成。
- 耗时信息：总耗时、首包等待时间；如果无法可靠区分首包，则显示已等待时间。
- 空返回提示：当上游成功返回但没有正文内容时，显示“模型返回为空”，同时保留 usage、模型和耗时信息。
- 错误详情：客户端可见错误优先展示网关返回的 `error.message`，并保留 HTTP 状态码、错误分类和上游原始摘要。
- 长时间等待提示：超过一个合理阈值后显示“模型仍在处理”，不自动判定失败；真正失败仍由请求错误、流式中断或超时事件决定。

交互要求：

- 发送按钮 loading 状态不改变输入区尺寸。
- 流式输出期间持续展示状态文本。
- 结束后把状态、耗时、usage 合并到 assistant 消息元信息中。
- 用户清空对话时同步清理当前状态、错误和上传文件临时状态。

### 日志定位

文件范围：

- `frontend/src/views/admin/Logs.vue`
- 必要时抽取轻量工具到 `frontend/src/utils/`，避免页面文件继续膨胀。

日志页保留当前表格，但增强筛选和分类可读性。

新增或强化的信息：

- 顶部错误分类摘要：展示当前筛选范围内的失败总数、上游反馈、网关配额、流式中断、上游响应异常。
- 快捷筛选保留“网关访问、网关配额、上游反馈、上游响应、流式中断”，并让当前激活状态更明显。
- 错误信息列展示可读摘要，完整错误通过 tooltip 或详情区域查看，避免表格被长 JSON 撑开。
- 重点错误分类需要中文说明，例如日 Token 限额、请求窗口限额、上游限流、上游临时不可用、上下文超限、流式空闲超时。

交互要求：

- 筛选条件变化后重置到第一页。
- 空结果时显示当前筛选条件下暂无日志，而不是空表格。
- 加载失败时展示后端返回的错误摘要；没有摘要时再使用“加载日志失败”。
- 表格字段继续优先使用已有原始透传字段：`api_name`、`log_type`、`billing_mode`、`user_agent`、`downstream_name`、`upstream_name`。

### Dashboard 与模型探测

文件范围：

- `frontend/src/views/admin/Dashboard.vue`
- `frontend/src/components/ModelProbeBoard.vue`
- `frontend/src/views/admin/ModelProbe.vue`
- `frontend/src/views/portal/ModelProbe.vue`

Dashboard 增加轻量“模型健康”区块，不替代现有请求趋势和失败分类。

模型健康区块展示：

- 健康、降级、离线通道数量。
- 平均探测耗时。
- 最近探测刷新时间。
- 一个跳转到模型探测页的入口。

模型探测页增强：

- 修正空状态和错误状态判断。空数据应显示“暂无通道数据”，接口失败或明确错误才显示“模型探测失败”。
- 增加搜索框，支持按上游名称、key 前缀、模型名称过滤。
- 增加状态筛选：全部、健康、降级、离线。
- 增加异常优先排序开关，默认让离线和降级通道更容易被看到。
- 通道卡片保持紧凑，模型列表过长时限制高度并允许滚动或折叠。
- 图表空数据时显示可读空态，不使用会误导的伪数据表达真实健康分布。

## 数据流

- 操练场继续调用当前门户侧模型列表和 Chat Completions 兼容接口，不新增协议字段依赖。
- 日志页继续通过 `adminApi.getLogs` 查询日志，不要求后端新增过滤字段。
- Dashboard 如果需要模型健康摘要，优先复用 `adminApi.getModelProbe` 返回的 `ModelProbeResponse`；不把模型探测数据混入 dashboard summary 协议。
- 模型探测页继续使用现有 `ModelProbeBoard` 数据结构，筛选和排序在前端完成。

## 错误处理

- 前端展示错误时遵循“后端结构化错误优先、原始消息兜底、通用文案最后兜底”。
- 不吞掉上游关键错误，例如配额达到上限、请求次数限制、上游限流、上下文超限、协议不支持、临时不可用。
- 对 JSON 错误体只做摘要展示，避免把大段原始内容直接塞进表格。
- 对空返回和失败返回做明确区分：空返回不是加载中，也不是成功生成了有意义内容。

## 测试要求

单元测试：

- `utils/playground` 覆盖空内容、usage 提取、流式状态文案、错误摘要解析。
- `ModelProbeBoard` 覆盖空状态、错误状态、状态筛选、搜索和异常优先排序。
- 日志展示工具覆盖错误分类中文说明和长错误摘要处理。

页面验证：

- 管理台 Dashboard 能正常加载并展示原有图表。
- 模型探测页在健康、降级、离线、空数据、接口失败场景下可读。
- 操练场在慢响应、空响应、错误响应和正常流式响应场景下都有明确状态。
- 日志页筛选、分页、空结果和加载失败行为正常。

部署验证：

- 构建前端。
- 运行现有全量测试。
- 在 `~/docker/chat-responses-codex` 部署目录做端到端 smoke，至少覆盖 health、front、models、chat、responses、claude、chat stream，以及管理台页面加载。

## 风险控制

- 先做前端展示和已有数据结构复用，不调整后端协议。
- 共享逻辑优先抽到小工具函数，避免页面组件继续堆积复杂判断。
- 所有新增筛选都在前端对当前接口返回数据处理，不改变接口查询语义。
- 页面文案使用中文，面向内部排障人员，避免营销式说明。

## 验收标准

- 用户打开 Dashboard 可以一眼看到模型健康概览，并能进入模型探测页定位异常通道。
- 用户在模型探测页可以按模型、上游、状态快速找到异常项。
- 用户在日志页可以按错误分类定位配额、限流、上游、协议和流式问题。
- 用户在模型操练场遇到慢模型、空返回或错误时，页面能清楚显示当前状态和失败原因。
- 不破坏现有 codex、opencode、claude code、Cline 插件、Hermes agent 等下游兼容链路。
