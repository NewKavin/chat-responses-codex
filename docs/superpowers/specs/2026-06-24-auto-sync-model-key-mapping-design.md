# 自动同步模型-key 映射设计

**目标：** 为所有活跃上游增加后台自动同步机制，按 key 轮询 `/v1/models` 并刷新 `api_key_models` 与 `supported_models`，以降低因模型映射过期导致的 499 和 `stream disconnected before completion`。

## 背景

当前系统已经支持在管理端手工探测上游模型，也支持在新增/更新上游时做一次性模型发现。但只要上游 key 的模型集合变化，旧映射就会逐步失真，路由层可能把请求发到不支持该模型的 key 上，最后表现为长尾超时、流式断开和大量 499。

这个问题的关键不是“有没有模型发现接口”，而是“模型-key 映射是否持续与上游真实能力对齐”。因此需要一个后台定时同步器，持续刷新每个 key 的模型列表。

## 核心设计

### 1. 共享模型发现逻辑

把“请求 `/v1/models` 并解析模型列表”的逻辑抽成共享模块，供以下场景复用：

- 管理端手工探测
- 管理端批量创建/更新上游
- 后台自动同步任务

这样可以避免不同入口各自实现探测逻辑后语义漂移。

### 2. 按 key 同步，而不是按 upstream 整体覆盖

一个 upstream 可能只有 1 个 key，也可能有多个 key，而且每个 key 支持的模型集合可能不同。

同步时要做到：

- 对每个 key 独立探测
- 探测成功时，用最新结果**替换该 key 旧的模型列表**
- 探测失败时，**保留该 key 旧的模型列表**
- 如果一个 upstream 里只有部分 key 成功，成功的 key 更新，失败的 key 继续沿用旧映射
- 如果这个 upstream 本轮所有 key 都失败，不做破坏性覆盖，保持原状态不变

### 3. `supported_models` 以“有效 key 映射”的并集为准

`supported_models` 不能继续作为手工填的宽泛白名单，否则会让路由层误判 upstream 仍支持某些模型。

同步规则是：

- 只要本轮至少有一个 key 成功，就用“成功更新后的 key 映射 + 失败 key 的旧映射”重新计算 `supported_models`
- 如果本轮所有 key 都失败，就保留原 `supported_models`

这样既能及时收敛过期模型，又能在局部故障时避免把 upstream 直接清空。

### 4. 同步周期独立配置

新增一个后端环境变量控制同步周期：

- `UPSTREAM_MODEL_KEY_SYNC_INTERVAL_SECONDS`
- 默认值：`900`

这个周期只控制后台同步，不影响前端模型探测页的自动刷新间隔。前端探测页仍然保留原有的浏览器轮询配置。

### 5. 启动即运行，之后定时运行

后台同步器在服务启动后立即执行一次，然后按配置间隔循环运行。这样服务刚启动时就能尽快纠正过期映射，不必等满一个周期。

### 6. 乐观并发保护

同步任务在发起探测时会先记录 upstream 的 `id`、`base_url` 和当前 key 列表快照。等探测结果返回并准备落库时，如果发现这个 upstream 在此期间已经被人工修改过，且 `base_url` 或 key 集合已经变化，则本轮跳过该 upstream，不把旧探测结果覆盖到新配置上。

这样可以避免后台同步和管理端手工编辑互相踩踏。被跳过的 upstream 会在下一轮定时同步中重新探测。

## 数据与状态

这次不新增数据库表，也不改 API 形状。只更新现有的上游配置内容：

- `api_key_models`
- `supported_models`
- `last_synced_at`

`last_synced_at` 只在本轮至少有一个 key 成功时更新，表示“确实有有效同步结果”的时间点。

## 错误处理

- 单个 key 探测失败不影响同 upstream 的其他 key
- 单个 upstream 失败不影响其他 upstream
- 全部 upstream 失败时只记日志，不中断服务
- 失败时保留旧映射，避免把路由层变成空洞状态

## 验收标准

- 单 key upstream 在模型变化后能被后台同步刷新
- 多 key upstream 中，成功的 key 被替换，失败的 key 保持旧映射
- 全部 key 失败时，原映射不被清空
- `supported_models` 与当前有效 key 映射保持一致
- 默认同步周期为 15 分钟
- 相关 env、docker compose、部署文档与测试断言全部更新
