# 上下游配置快速原子变更设计

## 背景

当前 PostgreSQL 中有 7,260 条 `usage_logs`。任意上游或下游配置变更都会调用
`PostgresStateStore::replace_state`；该函数在同步配置表后，又逐条执行全部历史日志的
`INSERT ... ON CONFLICT DO NOTHING`。生产日志显示，上游创建、更新、删除分别需要约
8.3 秒、7.9-8.6 秒、7.7 秒，下游更新约 9 秒，而配置列表查询仅需 9-20 毫秒。

配置持久化和能力探测也存在不必要的耦合。部分上游写入路径会等待有界能力探测队列，
最长可额外等待 5 秒；另一些更新路径仅依赖后台 reconcile。能力探测失败不应把已经提交
的配置写入报告为失败。

## 目标

- 在当前数据规模下，上游和下游的单条创建、更新、删除接口在 1 秒内完成。
- HTTP 成功只在配置数据库事务提交且内存路由同步完成后返回。
- 删除成功后，主配置记录和关联模型、allowlist、IP allowlist、能力 profile 不残留。
- 历史 usage log 作为审计记录保留，但不参与配置事务，也不因账号删除而删除。
- 配置变更触发的能力探测异步执行；探测队列满或暂时不可用不阻塞配置接口。
- 保持文件存储后端和 PostgreSQL 后端的配置语义一致。

## 非目标

- 不改变用户显式点击“获取模型”时对上游 `/v1/models` 的同步目录发现行为。
- 不改变多 key batch 接口对 key 和模型目录的验证协议；该网络操作仍受 30 秒管理端超时约束。
- 不删除或压缩历史 usage log。
- 不修改模型能力判定规则、探测用例或网关协议转换策略。

## 方案

### 1. 配置事务不再重放 usage log

`PostgresStateStore::replace_state` 只在一个事务中执行 `sync_config_tables`，随后提交。
删除其中对 `insert_usage_logs(&state.usage_logs)` 的调用。PostgreSQL usage log 已经由
`append_usage_logs` 独立批量追加；文件存储后端也已经在 `persist_config` 中显式写入空的
`usage_logs`，因此该变更统一了两个后端的职责边界。

配置事务不会删除 `usage_logs`，已有日志及其审计字段保持不变。

### 2. 数据库提交与内存切换保持原子

`AppState::mutate_persisted_state` 继续使用 `config_persist_lock` 串行化配置写入，但在创建
候选状态后保持 `inner` 锁，直到 `persist_config` 返回。数据库提交成功后，在没有新的
`.await` 取消点的情况下立即把候选配置字段写回内存，然后释放锁。

执行顺序为：

1. 获取配置写锁。
2. 获取内存状态锁，克隆并应用变更。
3. 持有内存状态锁执行配置持久化。
4. 持久化失败时保持原内存状态并返回错误。
5. 持久化成功后立即替换内存配置并返回。

移除 usage log 重放后，事务只操作少量配置行，短暂阻塞快照读取是可接受的。该顺序关闭了
“数据库已经提交，但请求取消发生在重新获取内存锁之前”的不一致窗口。

### 3. 能力探测改为非阻塞入队

配置创建、更新和自动同步仍生成按 `(upstream_id, runtime_model_slug, protocol)` 去重的
`ProbeJobBatch`，但发送改用非阻塞 `try_send`：

- 入队成功时由现有 worker 异步执行真实能力探测。
- 队列已满或 worker 暂不可用时记录结构化 warning，配置接口仍返回持久化结果。
- 现有每秒 reconcile 会重新发现缺失或指纹过期的 profile，并补充入队。
- 管理员显式调用手动探测接口时，仍保留队列不可用返回 503 的当前语义。

worker 的现有容量、去重、全局并发 2、单上游并发 1、单请求超时 20 秒均保持不变。

### 4. 删除一致性

PostgreSQL 配置事务继续以目标状态同步表：不存在于候选状态的上游或下游主记录会被删除，
外键级联负责删除以下关联配置：

- 上游：supported models、premium models、model request costs、dialect profiles。
- 下游：model allowlist、IP allowlist。

接口返回 204 后，还必须从内存路由、并发计数和能力快照中移除对应实体。usage log 不属于
配置残留：它保留原始上下游 ID 和名称用于历史审计。

### 5. 错误与可观测性

- 配置事务失败：返回 500，内存保持原配置，不显示成功提示。
- 输入无效或实体不存在：维持现有 400/404/409 行为。
- 能力探测入队失败：记录 upstream ID、任务数和队列状态，不记录 API key；不改变配置响应。
- HTTP access log 继续记录每个保存/删除请求的 `latency_ms`，用于部署前后对比。

## 测试

### PostgreSQL 持久化

- 构造 `PersistedState` 中存在但数据库尚不存在的 usage log，调用配置持久化后断言该日志
  不会被配置事务插入。
- 先通过 `append_usage_logs` 写入日志，再更新和删除配置，断言日志数量、内容和物理行不变。
- 删除上游后查询主表及四类关联表，断言均为零；删除下游后查询主表和两个 allowlist 表，
  断言均为零。

### 状态原子性

- 使用可控的测试 `StateStore` 暂停持久化，断言配置快照不会在提交前暴露候选状态。
- 持久化失败时断言数据库测试替身和内存快照均保持旧状态。
- 持久化成功后断言接口结果、内存快照和重新加载的数据库状态一致。

### 探测解耦

- 填满能力探测队列后创建或更新上游，断言配置写入立即成功，不再等待队列释放。
- worker 缺失或队列关闭时，断言配置仍成功，后台 reconcile 语义不影响持久化结果。
- 队列有容量时，断言完整 route batch 仍只入队一次。

### 生产验收

- 在保留当前 usage log 数据量的部署环境中创建、更新、删除临时上游和下游。
- 每个接口均低于 1 秒，并在返回后立即查询数据库和管理 API 验证一致。
- 验证历史 usage log 数量未减少，gateway 无 5xx、panic 或重启。
- 最终删除全部临时记录，并直接查询关联表确认无配置残留。

## 备选方案

### 为每类实体实现专用 SQL CRUD

专用 upsert/delete 可避免同步全部配置行，长期扩展性最好，但需要在 `StateStore` 中增加多组
后端接口，并维护文件存储与 PostgreSQL 两套实现。本次配置量很小，主要耗时来自 usage log
重放，因此暂不采用。

### HTTP 先返回、数据库后台提交

该方案能让界面立即结束 loading，但可能在进程退出、任务失败或客户端刷新时造成页面、内存
和数据库不一致，也无法保证删除后数据库没有残留，明确拒绝。

## 发布与回滚

该方案不修改数据库 schema。发布时构建新的 release 镜像，仅替换 gateway；部署后执行临时
实体的真实 CRUD 验收。回滚只需要恢复上一镜像，数据库结构和既有数据无需迁移或回退。
